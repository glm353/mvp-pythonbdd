## AC-02 Contractors sharing an email collapse to a single user   [SAF-473]
--------------------------------------------------------------------------------
Given: Two contractors (...0002 and ...0003) share the email bddpoc-dup@mailinator.com
(Beakon does not enforce a unique contractor email), per the Doc-003 STAGE 2 User
de-duplication logic (one winning record per normalized email)
When: I inspect business_fm_safezone.user in AWS Athena
Then: I expect exactly one winning user for the shared email — the lowest supplier_id
(uon_user_id ...0002), with is_supplier = true and no_of_sources = 1

AND: I expect no user row for uon_user_id ...0003 (the higher supplier_id lost the
de-dup; the original bug excluded BOTH, the fix keeps the lowest)

AND: I expect the surviving user ...0002 to still appear in the "All Contractors"
group (saf_group_id = 3pbmvtme6yuufgz5iwcilu2qn4)

## AC-03 A +615 number is dropped, but the contractor still onboards   [SAF-489/336]

(saf_group_id = 3pbmvtme6yuufgz5iwcilu2qn4)

## AC-04 An inactive contractor is excluded at the supplier boundary   [Doc-003 STAGE 1 is_active]
--------------------------------------------------------------------------------
Given: An inactive contractor (...0005, user_status = Inactive), excluded by the
Doc-003 STAGE 1 Supplier is_active = true filter
When: I inspect business_fm_safezone.supplier, .user and .user_group in AWS Athena
Then: I expect no row in business_fm_safezone.supplier for supplier_id ...0005

AND: I expect no row in business_fm_safezone.user for uon_user_id ...0005

AND: I expect no row in business_fm_safezone.user_group for uon_user_id ...0005

NOTE: This exclusion (rows removed by the STAGE-1 is_active filter) is distinct from
the SAF-235/SAF-524 supplier exception report, which FLAGS dirty-but-present rows
(e.g. invalid mobile, duplicate email) rather than removing them. The exception
report is out of scope for this feature.
