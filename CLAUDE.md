# mvp-pythonbdd — notes for Claude

Lean MVP distilled from **[poc-pythonbdd](https://github.com/glm353/poc-pythonbdd)** (ASP-1563), built
to the AI-driven tooling playbook (**ASP-1575 / ASP-1586**). Read [README.md](README.md) for the
user-facing overview and [WORKFLOW.md](WORKFLOW.md) for the Test Analyst ↔ DevOps split.

**What this repo does:** take selected **acceptance criteria** → have **AI (Claude on AWS Bedrock)**
author the behave `.feature` + steps via the **bdd-author skill** → drive them against an
**already-confirmed process chain (committed JSON, no generator)** → **print what the run would do in
AWS** (Step Functions triggers, Athena seed + assertion, teardown) as a summary and as
`out/execution-plan.json`. It **plans, it does not execute** — no boto3 calls to Step Functions/Athena.
(The PoC already ran this exact chain live on dev, all green — SESSION-NOTES 2026-06-18 / 06-22 — so the
printed plan is the verified sequence.)

## Two first-class moves (what makes this the MVP, vs the PoC)

1. **AI authoring is a real, re-runnable step.** `features/` and `features/steps/` ship **empty**.
   [author/author.py](author/author.py) loads [skills/bdd-author/SKILL.md](skills/bdd-author/SKILL.md)
   as the instruction, feeds the AC file + a context pack (the committed chain, the step vocabulary, the
   `ScenarioContext` API), calls **Claude Opus 4.x on Bedrock**, and writes the `.feature` + steps +
   `environment.py`. Don't pre-commit those generated files — running the author step IS the demo.
2. **Execution is planned, not performed.** [mvp_bdd/planner.py](mvp_bdd/planner.py) `PlanBackend`
   implements the backend shape (`seed`/`run_process`/`query`/`sql_table`) but **records the intended
   AWS action instead of doing it**. `query` returns `[]`, so `count=0=pass` assertions register as
   "would pass" — the planner describes intent; it does not validate live data.

## The one idea that must not be broken

**The "When" names zero processes.** The chain is resolved from two scenario endpoints: **source** ←
the Given (entity → [mvp_bdd/derivation.py](mvp_bdd/derivation.py) `ENTITY_SOURCES` → `Beakon.contractor`),
**target** ← the Then (schema of the asserted table → system token, e.g. `business_fm_safezone.user_group`
→ `safezone`). Resolution is **lazy**: `ScenarioContext.request_run()` (the When) only sets a flag; the
chain is loaded and walked on the **first Then** (`_ensure_chain_run`), using that Then's table to pick
the target. Don't refactor the When into hand-listing processes.

## Architecture (all logic in `mvp_bdd/`; the behave runner is thin and generated)

```
derivation.py   (source,target) from the scenario; ENTITY_SOURCES; logical↔env-qualified table helpers
                (schema_with_env, system_of_table, chain_key); fixture loaders.   [copied from the PoC]
chain.py        load_chain (committed artifact by source+target key) | run_chain (walk execution_order,
                hand each NON-source process to the backend). NO live generator.
assertions.py   compile_expectation / compile_rejection: a declared expected row → SQL in count=0=pass
                form (CAST("col" AS varchar) for Athena type-safety).            [copied from the PoC]
scenario.py     ScenarioContext = the Given/When/Then logic; lazy chain from the first Then.
planner.py      PlanBackend (records intended AWS actions) + print_plan/dump_plan + a --selfcheck.
__init__.py     dirs (chains/ and fixtures/ are at the REPO ROOT) + ENV_CODE="dev".

author/         author.py (Bedrock Converse), bedrock_config.py (env-config), environment.py.tmpl
                (deterministic lifecycle template copied to features/environment.py — NOT LLM-authored).
chains/         beakon-contractor__safezone.json — the confirmed chain (exec order + terminals + per-
                process {database_name, silver_table, process_type}).
fixtures/       test_contractor_dev.json — the real-dev-schema template the Given data table merges over.
acceptance-criteria/  contractor-safezone.acs.txt — the analyst input (real AC-NN / Given:When:Then).
skills/bdd-author/SKILL.md — the authoring instruction fed to Bedrock.
features/       GENERATED by author.py: contractor_safezone.feature, steps/contractor_steps.py, environment.py.
out/            execution-plan.json (gitignored).
```

## Gotchas / conventions (carried from the PoC)

- **`user` is a reserved word / quoted identifiers.** `PlanBackend.sql_table` emits
  `"schema_dev"."table"` (double-quoted, env-qualified) so the printed Athena SQL is valid.
- **Logical vs env-qualified names.** ACs/fixtures use *logical* schemas (`business_fm_safezone.user_group`);
  the chain artifact + plan use *env-qualified* (`business_fm_safezone_dev`). `schema_with_env`
  (default `ENV_CODE="dev"`) bridges them.
- **Planner is chain-generic — zero scenario knowledge.** `run_process(meta)` gets a process dict
  straight from the chain JSON; `seed(table, rows)` from the Given; `query(sql)` from the compiled Then.
  Swap the chain JSON or the ACs and the plan changes with no code change. Only genuine PLATFORM ids
  (common-sf ARN, workgroup `dev3`, region) are constants in `planner.py`, labelled "observed, not executed".
- **Real ACs aren't tidy.** Only some `[QA] SAF` stories carry GWT ACs (e.g. SAF-327, the `AC-NN`
  Athena-inspection template); others are Doc-003 links or sub-bugs with SQL. The AC file is written in
  that real template — see [WORKFLOW.md](WORKFLOW.md).

## Run / verify

```bash
.venv/Scripts/python -m mvp_bdd.planner --selfcheck    # offline: seed+4 SFN+asserts+teardown, JSON ok
.venv/Scripts/python author/author.py                  # Bedrock authoring -> fills features/ (needs SSO)
.venv/Scripts/python -m behave --dry-run features/contractor_safezone.feature   # 0 undefined
.venv/Scripts/python -m behave features/contractor_safezone.feature             # prints plan + writes JSON
```

`author.py --dry-run` builds and prints the prompt without calling Bedrock. Default model id is in
[author/bedrock_config.py](author/bedrock_config.py): `au.anthropic.claude-opus-4-8` (Opus 4.8 inference
profile, verified ACTIVE in account 484438948628 / ap-southeast-2 on 2026-06-26). Override
`BDD_AUTHOR_BEDROCK_MODEL` for a different account/region (e.g. the `global.` prefix) — list options with
`aws bedrock list-inference-profiles --region ap-southeast-2`.
