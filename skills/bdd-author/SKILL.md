# bdd-author — author a behave feature/steps from a QA acceptance criterion

DevOps uses this skill (run on AWS Bedrock by `author/author.py`) to translate a **Test Analyst's
acceptance criteria** into an **executable scenario**, consistently and to project convention.

**Human-in-the-loop:** the AI drafts; DevOps validates the `behave --dry-run`; the Test Analyst signs
off that the scenario faithfully captures the AC. The AI does not decide behaviour.

## Input you are given
- The analyst ACs (the `## AC-NN` blocks, each `Given:/When:/Then:/AND:`), which verify the deployed
  entities by *inspecting the Athena tables* against Doc-003. They are written in the analyst's voice
  ("When: I inspect the table in AWS Athena / Then: I expect ..."). Your job is to express that intent
  in the project's step vocabulary.
- The target tables + real columns/values (in the ACs and Doc-003).

## Recipe (AC -> scenario)
1. **Sample records -> Background.** Seed the analyst's sample population once with
   `Given the following Beakon contractors are seeded in dev` + the data table, then
   `When the SafeZone integration runs`. (The analyst's "When: I inspect the table" maps here: the run
   happens once; each scenario then inspects via its Then.)
2. **Each AC -> one Scenario.** The AC title becomes the scenario name; add a `# SAF-xxx` comment.
3. **Then/AND -> assertions.** For each `Then:`/`AND:` that names a table + expected row, write
   `Then the table "<schema.table>" has a row with` (inclusion) or `... has no row with` (exclusion)
   + a `| column | value |` table using the **real Doc-003 columns/values**. Use the terminal-check
   step where the AC asserts the chain endpoint.
4. **The When names ZERO processes** — never list integration processes; the chain auto-derives from
   the Then's target table.
5. **Reuse the step vocabulary** (below). Only add a new step if no phrase fits, and make it a thin
   wrapper over `mvp_bdd.scenario.ScenarioContext` (all logic stays in `mvp_bdd/`).

### Step vocabulary (reuse before adding) — bind to these EXACT phrases
- `Given the following Beakon contractors are seeded in dev` + a
  `| beakon_record_number | primary_email_address | mobile_number | user_status |` table
- `When the SafeZone integration runs`
- `Then the table "<schema.table>" has a row with` + a `| column | value |` table
- `Then the table "<schema.table>" has no row with` + a `| column | value |` table
- `And the auto-derived process chain ends at the gotcha terminal "<terminal>"`

### The step file binds to this ScenarioContext API (do not invent methods)
- `context.ctx.add_contractor_rows(rows, "contractor", template_fixture="test_contractor_dev.json")`
  — `rows` = `[dict(zip(context.table.headings, r.cells)) for r in context.table.rows]`
- `context.ctx.request_run()`  — the When
- `context.ctx.assert_table_row(table, parse_kv_table(grid))` / `assert_table_no_row(...)`
- `context.ctx.assert_terminal(terminal)`
  where `grid = [list(context.table.headings)] + [list(r) for r in context.table.rows]` and
  `parse_kv_table` is imported from `mvp_bdd.scenario`.

## Conventions that must hold
- **Assertions** compile to the `count=0=pass` form (`mvp_bdd/assertions.py`): string/number ->
  `CAST("col" AS varchar) = '<val>'`; `null` -> `IS NULL`; `true`/`false` -> `= TRUE/FALSE`.
- **Lifecycle:** all contractors seeded once + chain walked once (feature-scoped context in
  `features/environment.py`, generated by the author step). Every scenario asserts against that one walk.
- **This MVP PLANS, it does not execute.** The run prints the intended AWS calls (`out/execution-plan.json`)
  — never imply rows were validated live; the assertions describe intent.
- **Gherkin support:** behave allows `Feature`/`Background`/`Scenario`/`Scenario Outline`+`Examples`/
  `@tags`/`#` comments/`|` tables, but **NOT** `Rule` — never use it. Keep text ASCII-safe.

## Output + validation
Emit exactly two fenced code blocks, in this order:
1. ```gherkin  — the full `features/contractor_safezone.feature`
2. ```python   — the full `features/steps/contractor_steps.py` (thin wrappers over `ScenarioContext`)
Then DevOps runs `python -m behave --dry-run features/contractor_safezone.feature` (expect 0 undefined)
and hands to the Test Analyst for sign-off.
